API Security

When DevDocs integrates with backend APIs — for search indexing, user management, or content updates — those API calls must be secured properly. This page covers the key threats and countermeasures for API communication in a documentation platform context.

⚠ Never Expose API Keys Client-Side

Any secret embedded in JavaScript that runs in the browser is publicly visible to anyone who opens DevTools. Route all privileged API calls through a backend proxy or serverless function.

API Authentication Tokens

Bearer Tokens

The most common pattern for API authentication is the Authorization Bearer header: 

async function apiFetch(endpoint, options = {}) {
  const token = await getAccessToken(); // from secure storage

  const response = await fetch(`/api/v1/${endpoint}`, {
    ...options,
    headers: {
      'Authorization': `Bearer ${token}`,
      'Content-Type':  'application/json',
      ...options.headers
    }
  });

  if (response.status === 401) {
    handleUnauthorized();
    return null;
  }

  return response.json();
}

Token Refresh

Automatically refresh expired tokens before they cause request failures: 

async function getAccessToken() {
  let token = sessionStorage.getItem('access-token');
  const expiry = parseInt(sessionStorage.getItem('token-expiry') ?? '0', 10);

  if (Date.now() > expiry - 60_000) {
    // Refresh 60 seconds before expiry
    const data = await fetch('/auth/refresh', { method: 'POST', credentials: 'include' })
                        .then(r => r.json());
    token = data.accessToken;
    sessionStorage.setItem('access-token', token);
    sessionStorage.setItem('token-expiry', String(Date.now() + data.expiresIn * 1000));
  }

  return token;
}

Input Validation & Sanitization

All data sent to an API must be validated before dispatch. This prevents injection attacks and reduces server-side error rates. 

function validateSearchQuery(query) {
  if (typeof query !== 'string') throw new TypeError('Query must be a string');
  if (query.length === 0)        throw new Error('Query cannot be empty');
  if (query.length > 200)        throw new Error('Query exceeds maximum length');

  // Strip any HTML/script injections
  const div = document.createElement('div');
  div.textContent = query;
  return div.innerHTML.trim();
}

CORS Configuration

Cross-Origin Resource Sharing (CORS) headers must be set on the server. On the client side, always include credentials only when explicitly required: 

// Only send cookies/credentials when the API is same-origin or explicitly trusted
const response = await fetch('/api/v1/data', {
  credentials: 'same-origin'  // never use 'include' with a wildcard CORS origin
});

Server-Side CORS Headers

# Nginx — only allow your own domain
add_header Access-Control-Allow-Origin "https://docs.example.com";
add_header Access-Control-Allow-Methods "GET, POST, OPTIONS";
add_header Access-Control-Allow-Headers "Authorization, Content-Type";
add_header Access-Control-Max-Age 86400;

Rate Limiting

Protect your APIs from abuse and brute-force attacks with rate limiting. Implement it at the proxy or gateway layer, and handle 429 Too Many Requests gracefully in the client: 

async function apiFetchWithRetry(endpoint, options = {}, retries = 3) {
  const response = await apiFetch(endpoint, options);

  if (response?.status === 429 && retries > 0) {
    const retryAfter = parseInt(response.headers.get('Retry-After') ?? '2', 10);
    await new Promise(resolve => setTimeout(resolve, retryAfter * 1000));
    return apiFetchWithRetry(endpoint, options, retries - 1);
  }

  return response;
}

Dependency Security

Since DevDocs has zero runtime dependencies, the attack surface is minimal. However, if you add libraries, audit them regularly: 

# If using npm for build tools
npm audit
npm audit fix

# Check for known CVEs in specific packages
npx audit-ci --moderate
