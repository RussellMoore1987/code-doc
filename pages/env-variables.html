Environment Variables

Environment variables allow you to configure DevDocs deployments without hardcoding values in source files. They are the standard mechanism for separating configuration from code and adapting the same codebase across development, staging, and production environments.

ℹ Client vs. Server Context

DevDocs is a fully static site — there is no server process running at request time. Environment variables are useful during the build phase (if you add one) or when configuring the web server that serves the files. They cannot be read at runtime by browser JavaScript.

Common Configuration Variables

VariableDefaultDescription
DOCS_BASE_URL/Base URL path if deployed to a sub-path (e.g., /docs/).
DOCS_TITLEDocumentationSite title injected into <title> and header branding.
DOCS_VERSION—Displayed in the footer as the current documentation version.
DOCS_ANALYTICS_ID—Analytics tracking ID (e.g., GA4 measurement ID).
DOCS_SEARCH_ENDPOINT—URL of an optional server-side search API.
NODE_ENVdevelopmentBuild environment: development, staging, production.

Using .env Files

Manage environment-specific settings with a hierarchy of .env files: 

.env                  # Shared defaults (safe to commit if no secrets)
.env.local            # Local overrides (never commit)
.env.development      # Development environment
.env.staging          # Staging environment
.env.production       # Production environment

Syntax

# .env.production
DOCS_BASE_URL=/
DOCS_TITLE=MyProject Documentation
DOCS_VERSION=2.4.1
DOCS_ANALYTICS_ID=G-XXXXXXXXXX
NODE_ENV=production

Loading in Build Scripts

// build.mjs — example build script
import { config } from 'dotenv';
import { readFileSync, writeFileSync } from 'fs';

config({ path: `.env.${process.env.NODE_ENV ?? 'development'}` });

let html = readFileSync('index.html', 'utf8');
html = html.replace('__DOCS_TITLE__', process.env.DOCS_TITLE ?? 'Documentation');
html = html.replace('__DOCS_VERSION__', process.env.DOCS_VERSION ?? '');

writeFileSync('dist/index.html', html);

Passing Variables to Docker

# Inline
docker run -e DOCS_TITLE="My Docs" -p 8080:80 devdocs:latest

# From a file
docker run --env-file .env.production -p 8080:80 devdocs:latest

# In docker-compose.yml
services:
  docs:
    build: .
    env_file: .env.production
    ports:
      - "8080:80"

Environment Variables in CI/CD

GitHub Actions

# .github/workflows/deploy.yml
env:
  DOCS_TITLE: "MyProject Documentation"
  DOCS_VERSION: ${{ github.ref_name }}

jobs:
  deploy:
    steps:
      - run: echo "Deploying version $DOCS_VERSION"

Handling Sensitive Values

# Store secrets in GitHub Actions secrets, not env vars
- name: Build
  env:
    ANALYTICS_ID: ${{ secrets.DOCS_ANALYTICS_ID }}
  run: ./build.sh

Validating Required Variables

Fail fast in build scripts when required variables are missing, rather than producing a silently broken deployment: 

const required = ['DOCS_TITLE', 'DOCS_BASE_URL', 'NODE_ENV'];

for (const key of required) {
  if (!process.env[key]) {
    console.error(`Missing required environment variable: ${key}`);
    process.exit(1);
  }
}
