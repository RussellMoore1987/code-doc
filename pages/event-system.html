Event System

DevDocs ships with a lightweight publish/subscribe event bus that coordinates communication between decoupled subsystems — navigation, search, scroll spy, and third-party plugins — without creating hard dependencies between them.

ℹ Design Philosophy

The event system is intentionally minimal. It does not support event cancellation, bubbling, or priority queues. For complex orchestration needs, consider a state machine or a dedicated reactive library.

Built-in Events

The following events are emitted by the core application. Subscribe to them to react to lifecycle changes without patching internal functions.

Navigation Events

Event NamePayloadDescription
nav:before { navId, sectionId } Fired immediately before a navigation begins.
nav:after { navId, sectionId, success } Fired after navigation completes (or fails).
nav:page-loaded { url, navId } Fired when an HTML fragment is fetched and injected.

Search Events

Event NamePayloadDescription
search:open none Fired when the search overlay opens.
search:close none Fired when the search overlay closes.
search:query { query, resultCount } Fired after results are computed for a query.
search:select { navId, sectionId, query } Fired when the user selects a search result.

Theme Events

Event NamePayloadDescription
theme:change { theme: 'light' | 'dark' } Fired whenever the active colour scheme changes.

Subscribing to Events

Use EventBus.on(event, handler) to subscribe. The returned token can be passed to EventBus.off() for cleanup. 

// Subscribe
const token = EventBus.on('nav:page-loaded', ({ url, navId }) => {
  console.log(`Page loaded: ${url} (navId: ${navId})`);
});

// Unsubscribe later
EventBus.off(token);

For one-time subscriptions, use EventBus.once():

EventBus.once('search:open', () => {
  console.log('Search opened for the first time');
});

Publishing Custom Events

Plugins may publish their own namespaced events. Use a prefix to avoid collisions with core events: 

// Good — namespaced
EventBus.emit('myplugin:data-ready', { recordCount: 42 });

// Bad — could clash with core
EventBus.emit('ready', {});

Async Handlers

The event bus does not await async handlers. If you need sequential async execution, chain promises yourself: 

EventBus.on('nav:page-loaded', async ({ url }) => {
  // This runs fire-and-forget
  const analytics = await fetchAnalytics(url);
  trackPageView(analytics);
});

Error Handling

If a subscriber throws, the event bus catches the error, logs it to the console, and continues notifying remaining subscribers. One bad handler does not block the pipeline. 

EventBus.on('nav:after', ({ navId }) => {
  // If this throws, other handlers still run
  riskyOperation(navId);
});
⚠ Handler Cleanup

Always unsubscribe handlers that are registered in components or plugins that can be destroyed. Leaked subscriptions cause memory leaks and unexpected behaviour after teardown. Use the AbortController pattern or store tokens and call EventBus.off() in your cleanup routine.

Best Practices

  • Keep handlers small and focused — do one thing per handler.
  • Never mutate the payload object passed to your handler.
  • Use nav:before for analytics or permission checks, not to block navigation.
  • Test your handlers by emitting events in the browser console during development.
  • Document every custom event your plugin emits in its own README.
