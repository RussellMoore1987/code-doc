Authorization

Authorization controls what authenticated users are permitted to do and see. Unlike authentication (proving who you are), authorization enforces rules about access rights, content visibility, and allowed operations.

ℹ Prerequisites

This page assumes users are already authenticated. See the Authentication page for details on the login flow.

Role-Based Access Control (RBAC)

RBAC assigns permissions to roles, then assigns roles to users. It is the most common authorization model for documentation portals.

Role Hierarchy

RolePermissions
viewerRead public pages
memberviewer + read internal pages
editormember + edit page content via CMS
admineditor + manage users, settings, plugins

Checking Roles in Code

function hasRole(user, requiredRole) {
  const hierarchy = ['viewer', 'member', 'editor', 'admin'];
  const userLevel  = hierarchy.indexOf(user.role);
  const requiredLevel = hierarchy.indexOf(requiredRole);
  return userLevel >= requiredLevel;
}

// Usage
if (hasRole(currentUser, 'editor')) {
  showEditButton();
}

Page-Level Restrictions

Tag NavItems with a minimum required role, then filter them before the nav tree is rendered: 

// In NAV_DATA — add an optional 'requires' field
{
  id:       'billing',
  label:    'Billing Settings',
  page:     'pages/billing.html',
  requires: 'admin'   // Custom field
}

// In init, before buildNavTree():
function filterNavByRole(items, user) {
  return items
    .filter(item => !item.requires || hasRole(user, item.requires))
    .map(item => ({
      ...item,
      children: item.children ? filterNavByRole(item.children, user) : undefined
    }));
}

const visibleNav = filterNavByRole(NAV_DATA, currentUser);
elLeftNav.appendChild(buildNavTree(visibleNav));

Attribute-Based Access Control (ABAC)

ABAC grants access based on attributes of the user, resource, and environment rather than static roles. It is more expressive but also more complex to implement. 

function canAccess(user, resource, action, environment) {
  // Example: allow access only during business hours
  const hour = new Date().getHours();
  if (environment.requireBusinessHours && (hour < 9 || hour > 17)) {
    return false;
  }

  // Example: allow access if user's department matches resource owner
  if (resource.department && user.department !== resource.department) {
    return false;
  }

  return hasRole(user, resource.minimumRole ?? 'viewer');
}

Content-Level Gating

You can hide specific sections within a page by adding a data-requires attribute to any element and stripping unauthorized content in the afterPageLoad hook: 

<!-- In an HTML page fragment -->
<section data-requires="admin">
  <h2>Admin-Only Configuration</h2>
  <p>...</p>
</section>
// In app.js or a plugin
Hooks.after('page-load', ({ el }) => {
  el.querySelectorAll('[data-requires]').forEach(section => {
    const requiredRole = section.dataset.requires;
    if (!hasRole(currentUser, requiredRole)) {
      section.remove();
    }
  });
});

Audit Logging

Log authorization decisions for compliance and debugging. Fire an event on every access attempt (allowed or denied): 

Hooks.before('page-load', (url) => {
  const decision = canUserLoadPage(currentUser, url);

  auditLog({
    user:      currentUser.id,
    resource:  url,
    action:    'read',
    decision:  decision ? 'allow' : 'deny',
    timestamp: new Date().toISOString()
  });

  if (!decision) return false;
});
ℹ Compliance

If your documentation contains regulated information (HIPAA, PCI-DSS, SOC 2), audit logs must be tamper-proof and retained for the required period. Store them server-side, not in the browser.
