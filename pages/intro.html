Introduction

Welcome to DevDocs - a lightweight, generic documentation platform built for teams who need fast, structured technical content without heavy toolchain overhead. This guide walks you through the core concepts and helps you understand how everything fits together.

ℹ Note

This documentation site is served as a static project. No server-side rendering or framework is required. All you need is a basic local HTTP server or your IDE's Live Server extension.

What is DevDocs?

DevDocs is a three-column documentation layout inspired by modern developer portals. It features a collapsible left navigation tree, a dynamic content area, and a context-sensitive "On this page" outline on the right.

Key design goals include:

  • Keep dependencies to zero - pure HTML, CSS, and vanilla JavaScript.
  • Make it trivial to add or reorganise content by editing a single data structure in app.js.
  • Support light and dark themes with a single CSS variable system.
  • Provide a responsive layout that degrades gracefully on smaller screens. **Search Test**

Doc Link Usage

Use a[data-nav] links anywhere in page content to navigate to another page - or a specific section of another page - without a full page reload. The left sidebar will open the correct menu automatically.

Syntax

Page only:

<a href="#navId" data-nav="navId">Link text</a>

Page + section:

<a href="#navId#sectionId" data-nav="navId" data-section="sectionId">Link text</a>

In-page navigation only:

<a href="#sectionId" data-target="sectionId">Link text</a>

navId is the data-id data attribute of the nav item in the left hand menu, or page name in URL, e.g. image-processing in .../index.html#image-processing.

sectionId is the data-target data attribute of any heading on the target page, right hand menu, or in the URL as e.g. filters-effects in .../index.html#image-processing#filters-effects.

  • Go to the Image Processing page
  • Go directly to the Image Caching Optimization section of Image Processing
  • Go to deeply nested content like the Environment Setup -> testing-environment section.
  • Go to a section on this page theming section

Tooltip Usage

Add data-tooltip="…" to any element to give it a tooltip. Use the optional data-tooltip-position attribute to set a preferred direction. If the tooltip would overflow the screen, it flips to a better position automatically.

Syntax
<!-- Plain-text tooltip (default position: top) -->
<button data-tooltip="Tooltip text here">Hover me</button>

<!-- With preferred position -->
<button data-tooltip="Opens settings" data-tooltip-position="right">⚙</button>

<!-- HTML tooltip - write real HTML inside a <template> element -->
<template id="my-tip"><strong>Bold</strong> and <em>italic</em><br>Line two</template>
<button data-tooltip-template="my-tip">Hover me</button>

<!-- Available positions -->
top | top-left | top-right
bottom | bottom-left | bottom-right
left | right

Live examples - hover or focus each button:

top (default) top-left top-right bottom bottom-left bottom-right left right

Tooltips also work on non-button elements and support rich HTML content via data-tooltip-template:

data-tooltip ℹ Hover this link Long tooltip text ↑

Architecture Overview

The app consists of three files at the root level plus a pages/ directory that holds HTML fragments. The host page (index.html) loads styles.css and app.js; everything else happens at runtime.

File Structure

All project files live in a flat directory with no build step required:

doc/
├── index.html          ← Shell HTML (header, layout chrome, footer)
├── styles.css          ← All styles + CSS custom properties for theming
├── app.js              ← All behaviour: nav tree, Ajax, scroll spy, resize
└── pages/
    ├── intro.html      ← This page
    ├── installation.html
    ├── first-steps.html
    └── deep-topic.html

Content Loading

Each page in pages/ is an HTML fragment - just a <div class="content-body"> wrapper with your content inside. When a user clicks a left-nav link, app.js fetches that fragment via the Fetch API and injects it into the center column.

⚠ Important

Because pages are loaded via fetch(), you must serve this project through a local HTTP server. Opening index.html directly as a file:// URL will fail due to browser CORS restrictions. Use VS Code Live Server or run python -m http.server 8080 from the project folder.

Core Concepts

Navigation Data

All navigation is defined in a single NAV_DATA array in app.js. Each entry is a plain JavaScript object with id, label, and optionally page (path to an HTML fragment) or children (nested items). 

const NAV_DATA = [
  {
    id: 'getting-started',
    label: 'Getting Started',
    children: [
      { id: 'intro', label: 'Introduction', page: 'pages/intro.html' },
      { id: 'installation', label: 'Installation', page: 'pages/installation.html' },
    ]
  }
];

Theming

All colours and spacing tokens are defined as CSS custom properties on :root (light mode) and [data-theme="dark"] (dark mode). Switching themes is a single attribute change on <html>, which the browser handles in a single style pass.

Adding custom tokens

Open styles.css and add your token to both the :root and [data-theme="dark"] blocks, then reference it anywhere with var(--my-token).

Browser Support

DevDocs targets evergreen browsers. The following APIs are used without polyfills:

  • fetch() - page loading
  • IntersectionObserver - scroll spy
  • CSS Custom Properties - theming
  • DOMParser - HTML fragment parsing
  • localStorage - theme & sidebar persistence
✖ Not supported

Internet Explorer is not supported. No polyfills are provided or planned.
