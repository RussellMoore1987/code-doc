Message Queues

Message queues decouple the producer of a task from its consumer, enabling asynchronous processing, retry logic, and horizontal scaling. In a DevDocs deployment, queues are useful for sending notifications, rebuilding search indexes, processing feedback submissions, and triggering cache invalidations without blocking the main request.

ℹ When to Use a Queue

Use a message queue when a task is too slow to complete in an HTTP request-response cycle (> 500ms), could fail and needs retry logic, needs to fan out to multiple consumers, or must survive a server restart.

Basic Concepts

TermDefinition
ProducerThe service that creates and enqueues messages.
ConsumerThe service that reads and processes messages.
QueueA durable buffer that stores messages until consumed.
Dead Letter QueueA secondary queue that receives messages which exceed the retry limit.
AcknowledgementA signal from the consumer that a message was processed successfully.

BullMQ (Redis-backed Queue)

BullMQ is the most popular Node.js queue library, built on Redis:

Producer

import { Queue } from 'bullmq';
import { Redis } from 'ioredis';

const connection = new Redis(process.env.REDIS_URL);
const searchIndexQueue = new Queue('search-index', { connection });

// Enqueue a job when a page is published
export async function scheduleSearchReindex(pageId) {
  await searchIndexQueue.add('reindex', { pageId }, {
    attempts:       3,
    backoff: {
      type:   'exponential',
      delay:  2000
    },
    removeOnComplete: 100,
    removeOnFail:     50
  });
}

Consumer (Worker)

import { Worker } from 'bullmq';

const worker = new Worker('search-index', async (job) => {
  const { pageId } = job.data;
  console.log(`Reindexing page: ${pageId}`);

  const content = await fetchPageContent(pageId);
  await updateSearchIndex(pageId, content);

  return { reindexed: true };
}, { connection });

worker.on('completed', (job) => {
  console.log(`Job ${job.id} completed`);
});

worker.on('failed', (job, err) => {
  console.error(`Job ${job.id} failed: ${err.message}`);
});

AWS SQS

import { SQSClient, SendMessageCommand, ReceiveMessageCommand, DeleteMessageCommand } from "@aws-sdk/client-sqs";

const sqs  = new SQSClient({ region: 'us-east-1' });
const QUEUE_URL = process.env.SQS_QUEUE_URL;

// Produce
async function enqueue(message) {
  await sqs.send(new SendMessageCommand({
    QueueUrl:    QUEUE_URL,
    MessageBody: JSON.stringify(message),
    MessageAttributes: {
      Event: { DataType: 'String', StringValue: message.event }
    }
  }));
}

// Consume (polling loop)
async function poll() {
  while (true) {
    const { Messages } = await sqs.send(new ReceiveMessageCommand({
      QueueUrl:            QUEUE_URL,
      MaxNumberOfMessages: 10,
      WaitTimeSeconds:     20  // Long polling
    }));

    for (const msg of Messages ?? []) {
      await processMessage(JSON.parse(msg.Body));
      await sqs.send(new DeleteMessageCommand({
        QueueUrl:      QUEUE_URL,
        ReceiptHandle: msg.ReceiptHandle
      }));
    }
  }
}

Idempotency

In at-least-once delivery systems, the same message may be delivered more than once. Design consumers to be idempotent — processing the same message twice should produce the same result: 

const processed = new Set();

async function processMessage(message) {
  const key = `${message.event}:${message.data.pageId}:${message.timestamp}`;

  if (processed.has(key)) {
    console.log(`Skipping duplicate message: ${key}`);
    return;
  }

  await doWork(message);
  processed.add(key);
}
⚠ In-Memory Deduplication

The in-memory Set above is lost on restart. For production, store processed message IDs in Redis or a database with an appropriate TTL.
