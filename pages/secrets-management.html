Secrets Management

Secrets — API keys, database passwords, signing keys, and other sensitive credentials — must never appear in source code, client-side JavaScript, or version control history. This page covers secure patterns for managing secrets in DevDocs deployments of all sizes.

⚠ Common Mistake

Searching public GitHub repositories for patterns like apiKey = or SECRET_KEY regularly reveals thousands of accidentally committed credentials. Once committed, a secret is compromised — even after a history rewrite.

Environment Variables

The primary mechanism for injecting secrets into a running process is environment variables. They are not stored in files, are scoped to the current process, and are invisible to client-side code. 

# .env file (never commit this)
DOCS_API_KEY=sk_live_abc123xyz
DOCS_DB_URL=postgres://user:pass@host/db
DOCS_SIGNING_SECRET=supersecretvalue
# Load in your server/build script (Node.js example)
import 'dotenv/config';

const apiKey = process.env.DOCS_API_KEY;
if (!apiKey) throw new Error('DOCS_API_KEY is required');

Always Gitignore Secret Files

# .gitignore
.env
.env.local
.env.production
*.pem
*.key
secrets/

Secret Rotation

Secrets should be rotated regularly and immediately upon suspected compromise. Build your integration to handle rotation gracefully:

  1. Provision new credentials in the secrets store.
  2. Update the environment variable / secret reference without downtime (blue-green deploy or rolling restart).
  3. Revoke the old credentials only after the new ones are confirmed working.
  4. Update your audit log with the rotation event and operator identity.

Dedicated Secret Stores

For production deployments, use a dedicated secrets manager rather than flat .env files:

HashiCorp Vault

# Write a secret
vault kv put secret/devdocs api_key="sk_live_abc123"

# Read a secret (in deploy scripts)
export DOCS_API_KEY=$(vault kv get -field=api_key secret/devdocs)

AWS Secrets Manager

import { SecretsManagerClient, GetSecretValueCommand } from "@aws-sdk/client-secrets-manager";

const client = new SecretsManagerClient({ region: "us-east-1" });

async function getSecret(name) {
  const cmd = new GetSecretValueCommand({ SecretId: name });
  const response = await client.send(cmd);
  return JSON.parse(response.SecretString);
}

GitHub Actions Secrets

# .github/workflows/deploy.yml
- name: Deploy docs
  env:
    DOCS_API_KEY: ${{ secrets.DOCS_API_KEY }}
  run: ./deploy.sh

Scanning for Leaked Secrets

Integrate secret scanning into your CI pipeline to catch accidental commits before they reach a remote repository: 

# Install gitleaks
brew install gitleaks

# Scan the current repository
gitleaks detect --source . --verbose

GitHub's built-in secret scanning will alert repository owners when known credential patterns are pushed. Enable it under Settings → Security → Secret scanning.

Principle of Least Privilege

Every secret should carry the minimum permissions required for its purpose:

  • A read-only API key for fetching analytics does not need write permissions.
  • A deploy key for copying static files does not need database access.
  • Service accounts should be namespaced to a single application or service.
ℹ Audit Regularly

Review all active credentials quarterly. Revoke unused or overly-permissive keys. Most organizations accumulate dead credentials that represent unnecessary risk.
