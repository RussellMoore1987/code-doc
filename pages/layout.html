DevDocs GUI Features

This page documents the built-in UI behaviors available in the DevDocs shell: sidebar layout and resizing, responsive breakpoints, the search system, the light/dark theme toggle, and header navigation.

Sidebar Layout & Resizing

Both sidebars can be collapsed and resized interactively at any time:

  • Collapse - click the arrow icon (‹ / ›) in the sidebar header.
  • Expand - click the arrow icon in the narrow strip that appears when a sidebar is collapsed.
  • Resize - drag the divider between the sidebar and the center content area. Widths are persisted to localStorage so your preference is remembered across page loads.

Minimum Widths

The left and right sidebars each enforce a minimum width of 220 px. If you drag a sidebar below that threshold it will spring back to the minimum. You can adjust the MIN_SIDEBAR_WIDTH constant in app.js. The default widths on first load are 280 px (left) and 250 px (right); change DEFAULT_LEFT_WIDTH and DEFAULT_RIGHT_WIDTH to suit your content.

Persistence

Sidebar widths and collapse state are stored in localStorage under these keys:

  • devdocs-left-w / devdocs-right-w - pixel widths.
  • devdocs-left-collapsed / devdocs-right-collapsed - collapse state.

Clear these keys (e.g. via DevTools → Application → Local Storage) to reset to defaults.

Responsive Layout

DevDocs automatically collapses sidebars on narrow viewports to protect the center content area:

  • Below 1200 px - the right sidebar collapses automatically.
  • Below 800 px - the left sidebar also collapses.

On touch devices you can still expand either sidebar manually using the menu expand buttons (hamburger menus in the content area). The responsive collapse only triggers once on load; after that, user preference takes over.

Customizing Breakpoints

Edit the checkResponsiveCollapse() function in app.js and the matching media queries in styles.css to suit your content.

Search

The search system indexes every page listed in NAV_DATA from the app.js at startup. It will use these pages to search from. After typing into the search bar, it lets you select a result to jump directly to any section, then highlights all matches on the target page.

Opening & Closing Search

  • Click the search icon in the header actions area.
  • Keyboard shortcut - Ctrl+K (Windows/Linux) or ⌘+K (Mac) opens search from anywhere on the page.
  • Close - press Escape or click the × button, or click anywhere outside the search panel.

How the Index Is Built

On page load, buildSearchIndex() fetches every unique page URL found in NAV_DATA (in parallel). Each fetched document is parsed with DOMParser and broken into sections using headings (<h1>-<h6>). For each section the index stores:

  • The heading text (used as the result title).
  • The text content of all sibling elements that follow the heading, up to the next heading of equal or higher level (used for snippet generation and match counting).
  • The heading's id attribute, so a result click can scroll directly to that section.

Pages that cannot be fetched (e.g. 404) are skipped with a console warning and do not appear in results.

Querying

As you type, search is debounced by 300 ms to avoid excessive work. The query is validated before any search runs:

  • Maximum length is 25 characters.
  • Patterns that could indicate script injection (e.g. <script, javascript:, onload=, eval() are rejected and an error message is shown.

Matching is case-insensitive substring search across both the section title and its body text. Results are capped at 250 — if more are found the user is prompted to refine the query. To raise the cap, change the limit check in renderSearchResults() inside app.js.

Result List

Each result card shows the page name, section heading, and a text snippet with the matched term wrapped in a <mark> element. A badge appears when a section contains more than one match, showing the total count.

Keyboard Navigation in the Result List

  • ↓ / ↑ - move highlight through results.
  • Enter - navigate to the highlighted result (defaults to the first result if none is highlighted).
  • Escape - close search.

In-Page Highlights & Navigation

After you select a result, the target page loads and every occurrence of your search term is wrapped in a .search-highlight span. A floating navigation bar appears with:

  • Match counter - shows the current position, e.g. 3 of 11 matches.
  • Previous / Next buttons - step through all highlights one by one. Keyboard shortcuts ← / → also work while the floating navigation panel is open.
  • Clear button - removes all highlights and hides the navigation bar.

Highlights are cleared automatically whenever a new page is loaded or a new search is performed. The navigation bar's position adjusts when the right sidebar is collapsed so it stays visible.

Tag-Based Navigation

Any element on a content page can be tagged by adding a data-tags attribute with a comma-separated list of lowercase tag names: 

<section data-tags="api,reference">…</section>

At index time these tags are collected alongside the section title and a short snippet. An All Tags panel then appears at the bottom of the left sidebar, listing every unique tag found across all pages in the NAV_DATA from the app.js. Clicking a tag opens a filtered view of every section that carries that tag. The left sidebar tag panel is collapsible and its height is resizable (drag the handle) and persisted in localStorage under devdocs-left-tags-expanded and devdocs-left-tags-height.

Left Sidebar Tags

Clicking a tag in the left sidebar filters the content to show only sections that have that tag, making it easier to navigate related content across the documentation. You can click on another tag from the left hand panel to view that tag's related sections.

This tag-based navigation system allows for quick access to related content, improving the overall discoverability of documentation sections.

Right Sidebar Tags

The right-hand sidebar also includes page-level tag navigation, allowing you to quickly see every tag used on the current page. Selecting a tag filters the page content to display only the sections associated with that tag, similar to the behavior of the left-hand sidebar. While a tag filter is active, the right-hand sidebar's "On This Page" navigation and tag list are hidden to maximize visual space for the filtered content. To return to the normal page view, simply close the tag filter. The full page content, along with the "On This Page" navigation and page-level tags in the right-hand sidebar, will be restored.

Light / Dark Theme

Click the theme toggle button (sun/moon icon) in the header to switch between light and dark modes. The chosen theme is saved to localStorage under the key devdocs-theme and restored on the next page load. The data-theme attribute on <html> drives all color variables defined in styles.css.

Header Navigation

The top header contains a responsive navigation bar. When the viewport narrows and there is not enough room for all nav links, items are moved into an overflow dropdown triggered by a ≡ menu button. Below 550 px all nav items move to the overflow menu automatically. The overflow menu closes when:

  • A link inside it is clicked.
  • The user clicks outside the menu.
  • Escape is pressed (focus returns to the toggle button).

The redistribution of items is managed by updateHeaderNavigationOverflow() in app.js, which uses a ResizeObserver on the header inner element so it reacts instantly to any size change without polling.
