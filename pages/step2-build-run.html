Starting Your First Project

Let's create a new DevDocs project from scratch and get it running locally.

Project Setup

Create a new directory and set up the basic DevDocs structure: 

# Create a new project directory
mkdir my-documentation
cd my-documentation

# Create the basic structure
mkdir pages images
touch index.html app.js styles.css

# Copy DevDocs files (or download from repository)
# - index.html (main application shell)
# - app.js (navigation and functionality)  
# - styles.css (all styling)
# - pages/ (content directory)
# - images/ (assets directory)

Creating Your First Content Page

Create a welcome page to test your setup: 

<!-- pages/welcome.html -->
<div class="content-body">
  <h1 id="welcome">Welcome to My Documentation</h1>
  
  <p>
    This is your first DevDocs page! You can write any HTML content here,
    including headings, paragraphs, lists, and code blocks.
  </p>
  
  <h2 id="getting-started">Getting Started</h2>
  
  <p>Here are some things you can try:</p>
  
  <ul>
    <li>Edit this content and save to see changes</li>
    <li>Add more pages to the navigation</li>
    <li>Customize the theme and colors</li>
    <li>Test the search functionality</li>
  </ul>
  
  <div class="callout">
    <div class="callout-title">💡 Tip</div>
    <p>
      Remember to always wrap your content in a <code>&lt;div class="content-body"&gt;</code>
      element for proper styling and functionality.
    </p>
  </div>
</div>

Updating Navigation

Add your new page to the navigation in app.js: 

const NAV_DATA = [
  {
    id: 'getting-started',
    label: 'Getting Started', 
    children: [
      {
        id: 'welcome',
        label: 'Welcome',
        page: 'pages/welcome.html'
      }
    ]
  }
];

Running Locally

DevDocs requires a local HTTP server to function properly due to browser security restrictions on file:// URLs.

Development Server Options

VS Code Live Server (Recommended)

If you're using VS Code, the Live Server extension provides the easiest development experience:

  1. Install the "Live Server" extension in VS Code
  2. Right-click on index.html in the file explorer
  3. Select "Open with Live Server"
  4. Your documentation will open automatically at http://localhost:5500

Live Server automatically reloads the page when you save changes to any file.

Python HTTP Server

If you have Python installed (most systems do), you can use the built-in HTTP server: 

# Python 3 (recommended)
python -m http.server 8000

# Python 2 (legacy)
python -m SimpleHTTPServer 8000

# Then open http://localhost:8000 in your browser

Node.js Serve

If you have Node.js installed, you can use the `serve` package: 

# Install serve globally (one-time setup)
npm install -g serve

# Or use npx to run without installing
npx serve

# Then open http://localhost:3000 in your browser

Testing Core Functionality

Once your server is running, test these key features:

  • Navigation - Click items in the left sidebar to load content
  • Search - Use the search icon in the header to find content
  • Right Sidebar - Verify the "On this page" outline appears
  • Theme Toggle - Switch between light and dark themes
  • Responsive Design - Resize the browser window to test mobile layout

Development Workflow

Follow this workflow for efficient documentation development and maintenance.

Content Creation Cycle

  1. Plan Structure - Outline your documentation sections and pages
  2. Create Pages - Write HTML files in the pages/ directory
  3. Update Navigation - Add new pages to NAV_DATA in app.js
  4. Test Locally - Verify content loads and displays correctly
  5. Review Links - Check internal links and navigation flow
  6. Test Search - Ensure new content appears in search results

Content Best Practices

Follow these practices for maintainable, high-quality documentation:

HTML Structure

  • Use semantic headings - Start with h1, then h2, h3, h4 in order
  • Include unique IDs - Every heading should have a unique ID for deep linking
  • Structure content logically - Use sections, lists, and callouts appropriately
  • Keep HTML clean - Avoid inline styles, use CSS classes instead

Writing Guidelines

  • Start with overview - Begin each page with a clear introduction
  • Use active voice - "Click the button" vs "The button should be clicked"
  • Be concise - Break up long paragraphs into shorter, scannable sections
  • Include examples - Show both code examples and expected results

Preparing for Production

Before deploying your documentation, optimize it for production use.

Optimization Checklist

  • ✅ Minify CSS - Remove comments and whitespace from styles.css
  • ✅ Optimize Images - Compress PNGs/JPGs, convert to WebP if supported
  • ✅ Enable Compression - Configure gzip/brotli on your web server
  • ✅ Set Cache Headers - Configure appropriate cache expiration times
  • ✅ Test Search Index - Ensure search works with your full content
  • ✅ Validate HTML - Fix any HTML validation errors
  • ✅ Check Links - Verify all internal and external links work

Deployment Preparation

Prepare your files for deployment: 

# Create a production build directory
mkdir build
cp -r *.html *.css *.js pages/ images/ build/

# Optional: minify CSS (requires CSS minifier)
# csso styles.css --output build/styles.css

# Optional: optimize images (requires imagemin or similar)
# imagemin images/* --out-dir=build/images/

Deployment Options

DevDocs works on any static hosting platform. Popular options include:

  • GitHub Pages - Free hosting directly from your GitHub repository
  • Netlify - Drag-and-drop deployment with automatic HTTPS
  • Vercel - Fast global CDN with branch preview deployments
  • AWS S3 + CloudFront - Scalable hosting with CDN distribution

Quick Troubleshooting

Common Build & Run Issues

Problem: "Cross-Origin Request Blocked" errors in console.
Solution: Make sure you're using an HTTP server, not opening index.html directly.

Problem: Navigation shows "Loading..." indefinitely.
Solution: Check browser console for JavaScript errors and verify NAV_DATA syntax.

Problem: Search returns no results.
Solution: Wait for search index to build (check console), then try again.

Problem: Pages don't load when clicking navigation.
Solution: Verify page file paths in NAV_DATA match actual file locations.

Problem: Styling looks broken or inconsistent.
Solution: Check that styles.css is loading correctly and clear browser cache.
