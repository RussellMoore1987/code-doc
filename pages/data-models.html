Data Models & Schema

Understanding how DevDocs structures its internal data is essential for building integrations, writing custom plugins, and extending the platform's capabilities. This page documents all first-class data models, their fields, types, and validation rules.

ℹ Note

All models described here are plain JavaScript objects — there is no ORM or database layer. If you are integrating with an external data store, map these structures to your own schema.

NavItem Model

The NavItem is the fundamental building block of the navigation tree. Every entry in NAV_DATA conforms to this shape.

Fields

FieldTypeRequiredDescription
idstringYesUnique identifier used in URL routing and DOM queries.
labelstringYesHuman-readable display text shown in the sidebar.
pagestringNoRelative path to an HTML fragment. Omit for group nodes.
childrenNavItem[]NoNested items. A node cannot have both page and children.

Example

// Leaf node (has a page)
{
  id: 'installation',
  label: 'Installation',
  page: 'pages/installation.html'
}

// Group node (has children, no page)
{
  id: 'getting-started',
  label: 'Getting Started',
  children: [
    { id: 'intro', label: 'Introduction', page: 'pages/intro.html' }
  ]
}

Validation Rules

  • id values must be globally unique across all items at every depth level.
  • id should only contain lowercase letters, numbers, and hyphens (a-z0-9-).
  • A node with children must not also declare a page — the app treats such nodes as non-navigable group headers.
  • Depth is unlimited, but the UI is optimised for up to 4 levels of nesting.

Route State Model

The application maintains a currentRoute object inside the state singleton that describes the user's current position in the documentation. 

// state.currentRoute shape
{
  navId:     string | null,   // active NavItem id
  sectionId: string | null    // active heading id on the loaded page
}

This object is updated by navigateTo(), scroll spy events, and popstate handlers. Never mutate it directly — always call navigateTo() to trigger side effects (URL update, content load, right-nav highlight).

SearchIndexEntry Model

The search subsystem builds an in-memory index at load time. Each entry in searchState.index has the following shape: 

{
  url:     string,         // relative page path, e.g. "pages/intro.html"
  title:   string,         // text content of the first <h1> on the page
  navInfo: NavItem | null, // the matching NavItem from NAV_DATA
  sections: [
    {
      id:      string,  // heading element id (auto-generated if missing)
      title:   string,  // heading text
      content: string,  // raw innerHTML of sibling elements until next heading
      level:   number   // 1–6
    }
  ]
}

SearchResult Model

Each entry in searchState.results after a query looks like:

{
  type:          'section',
  page:          SearchIndexEntry,
  section:       SectionEntry,
  title:         string,   // heading text
  snippet:       string,   // HTML snippet with <mark> highlights
  navId:         string | null,
  sectionId:     string,
  instanceIndex: number    // 0-based occurrence index within the page
}

Application State Object

The global state object holds all mutable runtime data. Treat it as an internal singleton — plugins should read from it but avoid writing directly unless instructed. 

const state = {
  activeNavId:       string | null,
  scrollSpyObserver: IntersectionObserver | null,
  leftCollapsed:     boolean,
  rightCollapsed:    boolean,
  leftWidth:         number,   // pixels
  rightWidth:        number,   // pixels
  currentRoute:      RouteState | null,
  isNavigating:      boolean,
  abortController:   AbortController | null
};
⚠ Mutation Warning

Setting state.isNavigating = false manually to unblock a stuck navigation is an escape hatch. Prefer fixing the root cause (e.g., a rejected fetch) rather than forcing the flag.

Schema Best Practices

  • Keep id values stable — they are embedded in bookmarked URLs.
  • Use kebab-case for ids: api-reference, not apiReference.
  • Avoid id collisions between nav items and heading ids; the routing parser must disambiguate.
  • When adding pages programmatically, call buildNavTree(NAV_DATA) again and replace the contents of #left-nav to re-render the sidebar.
