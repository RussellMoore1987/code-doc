State Management

DevDocs uses a centralized, mutable state object rather than an immutable store. This keeps the implementation simple and avoids framework dependencies, while still providing a single source of truth for all runtime data.

ℹ When to Read This

This page is most relevant if you are building a plugin, writing integration tests, or trying to understand how the app responds to user interactions under the hood.

The State Object

All mutable runtime data lives in the state singleton declared at the top of app.js: 

const state = {
  activeNavId:       null,   // Currently displayed nav item id
  scrollSpyObserver: null,   // Active IntersectionObserver instance
  leftCollapsed:     false,
  rightCollapsed:    false,
  leftWidth:         280,    // px, persisted in localStorage
  rightWidth:        250,    // px, persisted in localStorage
  currentRoute:      null,   // { navId, sectionId }
  isNavigating:      false,  // Guards against re-entrant navigation
  abortController:   null    // AbortController for event listeners
};

Persisted State

Several properties are mirrored to localStorage so they survive page reloads:

State PropertyStorage KeyPersisted When
leftWidthdevdocs-left-wAfter drag or keyboard resize
rightWidthdevdocs-right-wAfter drag or keyboard resize
leftCollapseddevdocs-left-collapsedOn collapse (removed on expand)
rightCollapseddevdocs-right-collapsedOn collapse (removed on expand)
themedevdocs-themeOn every theme toggle

Reading State

Read directly from the state object. Because state is a module-level variable, any code in the same script context can access it: 

// Is the right sidebar open?
if (!state.rightCollapsed) {
  adjustLayout();
}

// What page are we on?
const { navId, sectionId } = state.currentRoute ?? {};

Writing State

Prefer using the provided functions rather than mutating state directly, because the functions also apply DOM changes and persist to localStorage:

Desired ChangeUse This Function
Collapse left sidebarcollapseLeft()
Expand left sidebarexpandLeft()
Collapse right sidebarcollapseRight()
Expand right sidebarexpandRight()
Navigate to a pagenavigateTo(navId, sectionId)
Switch themeapplyTheme('light' | 'dark')
⚠ Direct Mutation Risks

Writing to state.leftCollapsed = true without calling collapseLeft() will desync the state object from the DOM. The sidebar will still be visible even though state.leftCollapsed says otherwise. Always use the API functions.

Resetting State

To perform a full reset — for example, in integration tests — call cleanup() followed by a fresh init(): 

// Tear down
cleanup();

// Re-initialise (re-reads localStorage)
init();

To clear persisted preferences only (without re-initialising), clear the relevant localStorage keys: 

localStorage.removeItem('devdocs-left-w');
localStorage.removeItem('devdocs-right-w');
localStorage.removeItem('devdocs-left-collapsed');
localStorage.removeItem('devdocs-right-collapsed');
localStorage.removeItem('devdocs-theme');

Search State

The search subsystem uses its own searchState object to avoid cluttering the main state singleton: 

const searchState = {
  isOpen:            false,
  index:             [],       // Array<SearchIndexEntry>
  currentQuery:      '',
  selectedResult:    -1,
  results:           [],       // Array<SearchResult>
  hasValidationError: false,
  currentHighlights: [],       // Array<HTMLElement>
  currentMatchIndex: -1
};

Reset search state programmatically by calling closeSearch(), which zeroes most of these fields, or clearPageHighlights() for highlight-specific state.
