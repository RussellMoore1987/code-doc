Deep Topic

This page demonstrates the deepest topics in the DevDocs configuration system — custom theming, navigation model extension, scroll-spy tuning, and build optimisation. It also serves as a living example of all supported content elements.

Custom Theming

DevDocs uses a single set of CSS custom properties (variables) to control every colour and visual token. Override them to match your brand without touching any existing CSS rules.

Token Anatomy

Tokens are grouped into seven categories inside styles.css:

  • Backgrounds — body, header, sidebar, content area, code blocks.
  • Text — base, muted, links, nav states, code.
  • Borders — default border and focus ring.
  • Accents — primary and light accent colours.
  • Callout colours — per variant (info, warning, danger).
  • Resize & interactive elements — handles, hover, drag states.
  • Navigation decoration — indent lines, active indicator bar.

Overriding Tokens

Add a <style> block at the bottom of index.html (or create a theme.css and link it after styles.css):

/* theme.css — custom brand overrides */
:root {
  --accent:       #7c3aed;   /* purple primary */
  --accent-light: #ede9fe;
  --nav-active-bar: #7c3aed;
  --callout-border: #7c3aed;
  --bg-callout:   #f5f3ff;
}

[data-theme="dark"] {
  --accent:       #a78bfa;
  --accent-light: #2e1065;
  --nav-active-bar: #a78bfa;
  --callout-border: #a78bfa;
  --bg-callout:   #1e1030;
}

Replacing the base font

The body font is set via a system font stack. To use a custom typeface, load it from Google Fonts or a local file, then override the font-family on body: 

<link rel="stylesheet"
      href="https://fonts.googleapis.com/css2?family=Inter:wght@400;600;700&display=swap" />

<style>
  body { font-family: 'Inter', sans-serif; }
</style>

Extending the Navigation Model

The NAV_DATA array in app.js is the single source of truth for the entire left-sidebar tree. The renderer is recursive, so any depth is technically supported (though four levels is the practical limit for usability).

Nav Item Shape

// Leaf item  (has a page, no children)
{ id: 'unique-id', label: 'Display Name', page: 'pages/my-page.html' }

// Group item  (has children, may or may not have a page)
{
  id: 'group-id',
  label: 'Group Label',
  children: [
    { id: 'child-a', label: 'Child A', page: 'pages/child-a.html' },
    { id: 'child-b', label: 'Child B', page: 'pages/child-b.html' },
  ]
}

Groups with their own page

A group can have both children and a page property. Clicking saves the active state and loads the page; the toggle arrow expands/collapses the children independently. This is useful for section overview pages: 

{
  id: 'advanced-overview',
  label: 'Advanced Topics',
  page: 'pages/advanced.html',   // ← loads on click
  children: [
    { id: 'config',  label: 'Configuration', page: 'pages/config.html' },
    { id: 'plugins', label: 'Plugins',        page: 'pages/plugins.html' },
  ]
}

Scroll Spy Tuning

Scroll spy is implemented with an IntersectionObserver rooted to the center scroll container. The default rootMargin is 0px 0px -70% 0px, which means a heading triggers the active state as soon as its top edge enters the top 30 % of the scroll area.

Changing the Threshold

Find the setupScrollSpy() function in app.js and adjust the rootMargin: 

const observer = new IntersectionObserver(callback, {
  root:       elContentScroll,
  rootMargin: '0px 0px -60% 0px',  // ← increase % to trigger earlier
  threshold:  0
});

Sticky header offset

The CSS rule scroll-margin-top: 68px is applied to all heading IDs to account for the sticky header. If you change the header height, update this value in styles.css: 

.content-body h1[id],
.content-body h2[id],
.content-body h3[id],
.content-body h4[id] {
  scroll-margin-top: 68px; /* ← match your header height */
}

Ajax Page Loading Details

Pages are fetched with the native fetch() API. The HTML response is parsed with DOMParser and the <body> content is extracted before injection, so full HTML documents, fragments, and partial templates all work equally well.

Error Handling

If a fetch fails (404, network error, CORS), the center column displays an inline error card with the status code and a remediation hint. The left nav active state is still set so the user knows which link failed.

⚠ Security Note

Scripts inside fetched fragments are stripped before injection. Only trusted, local HTML fragments should be referenced in NAV_DATA. Do not point pages at arbitrary third-party URLs.

Pre / Post-Load Hooks

To run your own code before or after a page loads, add it to the loadPage() function in app.js. The function is async and returns a Promise: 

async function loadPage(url) {
  // Pre-load hook
  document.dispatchEvent(new CustomEvent('page:before-load', { detail: { url } }));

  // ... existing fetch logic ...

  // Post-load hook  (inside the try block, after injection)
  document.dispatchEvent(new CustomEvent('page:loaded', { detail: { url } }));
}

Performance & Optimisation

Because there is no build step, optimisation is straightforward:

  • Minify assets — run styles.css and app.js through any minifier (e.g., terser, cssnano) before deploying to production.
  • Compression — enable gzip or Brotli on your web server; HTML and CSS compress exceptionally well.
  • Caching — set aggressive Cache-Control headers on page fragments once content is stable.
  • Image optimisation — use webp format and the <picture> element for responsive images.

Prefetching Pages

For instant navigation, add <link rel="prefetch"> hints in the document head for pages the user is likely to visit next. Browsers will download them in the background during idle time: 

<!-- index.html <head> -->
<link rel="prefetch" href="pages/installation.html" />
<link rel="prefetch" href="pages/first-steps.html" />

Bundle Size

The un-minified assets weigh approximately:

  • styles.css — ~9 KB
  • app.js — ~6 KB
  • index.html — ~5 KB

After minification and gzip compression, total transfer for the shell is well under 10 KB. Page fragments are loaded on demand and are typically a few KB each.

Deploying to Production

DevDocs can be hosted on any static hosting platform: GitHub Pages, Netlify, Vercel, Cloudflare Pages, AWS S3 + CloudFront, or a traditional web server.

GitHub Pages

  1. Push your project to a GitHub repository.
  2. In repository Settings → Pages, set the source branch to main and the folder to /root.
  3. GitHub Pages will serve index.html at https://<username>.github.io/<repo>/.
✖ Trailing-slash paths

On GitHub Pages, make sure all page paths in NAV_DATA use a relative path from the repository root (e.g., pages/intro.html), not an absolute path starting with /, or your fetches will fail.

Netlify & SPA Redirects

Since DevDocs is not a single-page app with client-side routing — all navigation is internal fetch-based, not URL-based — you do not need a _redirects file or fallback rule. Serve the project as-is.
