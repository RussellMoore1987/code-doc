Lifecycle Hooks

Lifecycle hooks give you precise control over the application's initialization and teardown sequence. By hooking into specific phases you can run setup code at the right moment without modifying core files.

ℹ Hook vs. Event

Hooks fire synchronously during a specific phase and can block it (by throwing). Events fire asynchronously after a phase completes and cannot block it. Use hooks for critical setup; use events for side effects and analytics.

Initialization Phases

The init() function runs in this order when DOMContentLoaded fires:

  1. beforeInit — runs before any setup.
  2. AbortController created.
  3. Theme applied.
  4. Sidebar widths applied.
  5. Nav tree built and appended.
  6. Collapse state restored.
  7. Resize handles wired.
  8. Responsive collapse checked.
  9. Routing initialized.
  10. Search initialized.
  11. Copy buttons set up.
  12. afterInit — runs after all setup.
Hooks.before('init', () => {
  console.log('App is about to initialize');
});

Hooks.after('init', () => {
  console.log('App fully initialized');
  trackPageLoad();
});

Page Lifecycle Hooks

Every time a content page is fetched and rendered, the following hooks fire in order:

beforePageLoad

Fires before the fetch request is made. Receives the target URL. Returning false from a synchronous hook will cancel the load. 

Hooks.before('page-load', (url) => {
  if (url.includes('restricted')) {
    return false; // Abort
  }
});

afterPageLoad

Fires after the fragment has been injected into the DOM but before scroll spy and right-nav are built. Useful for post-processing content. 

Hooks.after('page-load', ({ url, el }) => {
  // el is the #content-body element
  el.querySelectorAll('a[href^="http"]').forEach(a => {
    a.setAttribute('target', '_blank');
    a.setAttribute('rel', 'noopener noreferrer');
  });
});

afterRightNavBuild

Fires after the right-side outline is generated from page headings. Use this to inject custom entries or badges. 

Hooks.after('right-nav-build', ({ navEl, headings }) => {
  // Add a "Back to top" entry
  const li = document.createElement('li');
  li.innerHTML = '<a href="#" class="right-nav-link">↑ Back to top</a>';
  navEl.appendChild(li);
});

Cleanup Hooks

beforeCleanup fires before the app tears itself down on beforeunload and pagehide: 

Hooks.before('cleanup', () => {
  // Flush pending analytics events
  analytics.flush();
});

Registering and Removing Hooks

Hooks API

// Register
const token = Hooks.before('page-load', myHandler);
const token2 = Hooks.after('init', myOtherHandler);

// Remove
Hooks.remove(token);
Hooks.remove(token2);

One-Time Hooks

// Fires exactly once, then auto-removes
Hooks.once('after', 'init', () => {
  console.log('First init complete');
});

Hook Execution Order

Multiple handlers registered for the same hook run in registration order. If any handler throws, the remaining handlers in that slot still run (errors are caught and logged), but a hook that explicitly returns false will halt the chain.

⚠ Synchronous Requirement

before hooks that can return false must be synchronous. Async before hooks are awaited but their return value is ignored for cancellation purposes. Use the middleware pipeline if you need async cancellation.

Best Practices

  • Use afterInit for plugin bootstrapping, not global side effects.
  • Keep beforePageLoad fast — it blocks navigation.
  • Always store hook tokens and remove them in your cleanup routine.
  • Don't rely on hook ordering across different plugins; use the EventBus for coordination.
