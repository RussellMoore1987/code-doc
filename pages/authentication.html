Authentication

Out of the box, DevDocs is a public documentation site with no authentication layer. This page explains how to add user authentication — restricting access to internal docs, staging environments, or paid content — using common patterns that integrate cleanly with the existing architecture.

⚠ Security Notice

Client-side authentication guards are convenience features, not security boundaries. Sensitive content must be protected at the server or CDN layer. Never rely solely on JavaScript to hide confidential information.

Authentication Patterns

HTTP Basic Auth

The simplest approach: configure your web server (Nginx, Apache, or a CDN edge function) to require HTTP Basic Auth before serving any files. No code changes to DevDocs are needed. 

# Nginx example
location /docs/ {
  auth_basic "Documentation";
  auth_basic_user_file /etc/nginx/.htpasswd;
}

JWT Token Validation

For single-page app style authentication, intercept navigation in a beforePageLoad hook and validate a JWT stored in sessionStorage: 

Hooks.before('page-load', (url) => {
  const token = sessionStorage.getItem('auth-token');

  if (!token || isTokenExpired(token)) {
    // Redirect to login, cancel page load
    window.location.href = '/login?return=' + encodeURIComponent(url);
    return false;
  }
});

function isTokenExpired(token) {
  const payload = JSON.parse(atob(token.split('.')[1]));
  return Date.now() / 1000 > payload.exp;
}
⚠ Token Storage

Avoid storing JWTs in localStorage — they are accessible to any JavaScript on the page, making them vulnerable to XSS. Prefer sessionStorage or HttpOnly cookies set by your backend.

SSO via OAuth 2.0

For enterprise environments, integrate with an Identity Provider (IdP) such as Okta, Azure AD, or Auth0. The authentication flow works as follows:

  1. User visits the docs URL without a valid session cookie.
  2. A server-side edge function redirects to the IdP authorization endpoint.
  3. After successful login, the IdP redirects back with an authorization code.
  4. The edge function exchanges the code for tokens and sets a secure session cookie.
  5. Subsequent requests are validated against the session cookie.

Session Management

Track session expiry in DevDocs by registering a periodic check: 

const SESSION_CHECK_INTERVAL = 5 * 60 * 1000; // 5 minutes

setInterval(() => {
  if (!isSessionValid()) {
    showSessionExpiredBanner();
  }
}, SESSION_CHECK_INTERVAL);

function isSessionValid() {
  return document.cookie.includes('session=');
}

Session Expired Banner

function showSessionExpiredBanner() {
  const banner = document.createElement('div');
  banner.className = 'callout warning';
  banner.innerHTML = `
    <div class="callout-title">Session Expired</div>
    <p>Your session has expired. <a href="/login">Log in again</a> to continue.</p>
  `;
  document.querySelector('#content-body').prepend(banner);
}

Displaying User Identity

Once authenticated, you can display the logged-in user's name in the header. Inject this after init() completes: 

Hooks.after('init', async () => {
  const user = await fetchCurrentUser();
  if (user) {
    const el = document.querySelector('.header-brand');
    const badge = document.createElement('span');
    badge.className = 'user-badge';
    badge.textContent = user.name;
    el.appendChild(badge);
  }
});

Role-Based Access Control

Restrict specific navigation sections to users with certain roles by filtering NAV_DATA before the nav tree is built: 

Hooks.before('init', async () => {
  const user = await fetchCurrentUser();
  const roles = user?.roles ?? [];

  // Remove admin section for non-admins
  if (!roles.includes('admin')) {
    const idx = NAV_DATA.findIndex(s => s.id === 'admin');
    if (idx !== -1) NAV_DATA.splice(idx, 1);
  }
});
ℹ Defence in Depth

Filtering the nav hides links from non-admin users but does not prevent direct URL access. Always enforce role checks on the server side for any restricted content.
